Completed Demo's Ready for Use!

Here are demo's created by the imapex team that are ready for wide spread use. These demo's may have continued development and enhancement so check back often.

  • MyHero
  • http://myhero.apps.imapex.io
  • hpreston/myhero_demo
  • Hank Preston
  • MyHero is a very simple sample application developed using a "Cloud Native" design method to be used for demonstration and education purposes. It is composed of several microservices, each leveraging standard API interaction. The minimal deployment of MyHero consists of just three services, and when deployed with all features, it can be as many as ten different services.

  • ioxduino
  • imapex/ioxduino
  • Hank Preston
  • IOx Proof of Concept to integrate a Cisco Router with an Arduino Microcontroller.
  • AppRegistry
  • http://appregistry.apps.imapex.io
  • imapex/appregistry
  • Kevin Corbin
  • This is a simple frontend for APIC-EM which allows business owners to register applications in a portal with a minimal amount of technical details about the application, and transfer that information to APIC-EM to be consumed in EasyQos policies

  • ACI Migration Wizard
  • imapex/acimigrate
  • Kevin Corbin & Chad Peterson
  • A migration wizard which will migrate VLAN configuration from Legacy Nexus environments to an ACI fabric

Demo's Currently In Development

Here are the demo's that are currently in development by the imapex team.


  • GADD Project: "Gomorrah Automated Distributed Defense"
  • imapex/gadd
  • Chet Puskarz & Ryan Shoemaker
  • Using Spark and NFVIS along with ACI and NGIPS* The idea of the demo is based on an event/alert in the datacenter, dynamically deploy a FW at offending remote brach. Phase 1 will be simulating an event/alert without an IPS.

  • Event Driven QoS
  • imapex/ed-qos
  • Steve Luzynski & Matt Brainard
  • Event Driven QoS is a simple demo application illustrating programmatic control of APIC-EM to change QoS parameters in an attached network. This is driven by an external event. In the initial implementation, the external event source is changes in the weather; other event sources are planned.

  • DC Jeeves
  • imapex/dcjeeves_demo
    imapex/dcjeeves_app
    imapex/dcjeeves_spark
    imapex/dcjeeves_ucsd
  • Mike Tokarz & Russel Wilkinson
  • DC Jeeves is your adaptable datacenter butler. His goal is to make your life easier by providing an interactive interface through tools like Spark & Tropo and turning your requests into actionable functions in your datacenter. Jeeves, provision a server for me!

  • IOT Notification Agent
  • Kevin Corbin & Chris Bogdon
  • This application serves as a generic event notification engine for IOT enabled routers. The engine will react to any external event and perform some type of notification.

  • CMX Spark Alert (csalert)
  • imapex/csalert
  • Bryan Adams & Paul Lewis
  • The goal is to create an application that will take location-based event triggers from CMX and output a Spark message to a specific room or user. The CMX input could be device tracking, congestion thresholds, geo-fencing alerts, etc.
    Our use case is a retail store that will detect shopper congestion (by tracking client count on a nearby AP) and send an alert message to the manager or employee in that department so that they can assist the shoppers.
    We would like to keep the app modular to adjust both the CMX inputs and the Spark output.
  • Time Data Path (TDP)
  • /imapex/time-data-path
  • Bryan Byrne & Ben Flowers
  • Time Data Path (TDP) leverages APIC-EM Path Trace to identify the path a flow takes through the network. Once the flow has been identified TDP uses either the APIC-EM API or NX-API to pull interface statistics and plot statistics over time using an OpenSource cachet database.
  • Spark Trak
  • Tim Wittbrod & Blair Hicks
  • Spark Trak is a professional collaboration application leveraging Cisco Spark to provide secure communication. Intended to be used by health care professionals, laywers, insurance agents, etc - any professional that needs to share information pertaining to a client. By entering into Spark Room, the professionals are able to discuss and access confidential records. While in the Spark Room, they are able to initiate actions relevant to the client - order procedures, prescribe medication, set appointments. Upon exiting the Spark room, the collaboration session is archived and can generate a billing record for the client.
